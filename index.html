3. Project Notes

Overall

Project Structure

There are established rules for project structure, although not all repositories, especially older ones, fully meet these standards. However, the ideal structure is set for convenience and should include the following folders:

  • scripts: Contains all the project's scripts used for running tasks, processing data, and automation.
  • docs or documentation: A folder that holds all the additional project-related documents, including any guides, manuals, or technical specifications.
  • logs: This folder should store output logs, which are generated by the scripts or the program to record its operation or errors.
  • sips: Contains submission packages (SIPs) created for the project. These are generally packaged files ready for submission to systems like Rosetta.
  • assets: Stores input data files that the project relies on, such as images, PDFs, or other resources.
  • database (if applicable): Used for storing any database files or related configurations that the project interacts with.
  • tests Ideally, tests should be added for each functions. In practice, only for some of them.
  • .github/workflows/ci.yml: This folder is for CI/CD pipelines. Ideally, each project should include a GitHub Actions workflow configuration file here to automate testing and deployment processes.

Best Practices

Current best practices for project management and script organization involve the following guidelines:

  • Main Script Storage: GitHub:

    The primary storage location for all final and version-controlled scripts is GitHub. Each project should have a corresponding repository on GitHub where the clean and final versions of the scripts are maintained. This ensures easy collaboration, version control, and code sharing.

  • Script Storage Locations:
    • Set working versions of the scripts are also kept on two additional servers:
      • Cascade (Y): A shared server where most of the scripts are currently stored for local execution and quick access.
      • G server: Another shared server that holds additional project scripts and data.
    • There is no dedicated server for running scripts. For full automation, container servers could be leveraged in the future to ensure proper isolation and scalability.
  • Use of Cascade Server (LD_working):

    Currently, all the projects are set on the Cascade server under the LD_working section. While this section provides cost-effective storage with a 30-day retention period, it is not ideal for long-term script storage or projects requiring a full history.

  • Project Folders and Scripts:

    Each project folder often contains more scripts than strictly required. This includes scripts for testing, experimentation, and cleaning. However, the clean and final version of the scripts can be found in the Project_name_git folders. These folders are synced with GitHub repositories and serve as the main point of communication with Git, although they are not perfectly aligned with best practices.

  • Git Integration:

    All projects use Git for version control, and all communication with Git repositories is set from the Project_name_git folders. Each project should ideally include only essential scripts and follow a structured commit history to ensure smooth collaboration and version management.

Tests and CI

Each project must have unit tests covering critical functionality. Continuous Integration (CI) should be used to automate testing on every push to the repository.

Podcasts

The Podcasts project involves automating the ingestion and processing of podcast files. Ensure proper integration with Google Sheets and Google API for metadata handling.

Issuu

The Issuu project collects PDFs and images from the Issuu platform. Two workflows are used for this project, both relying on web scraping tools like Selenium and Geckodriver.

LINZ

The LINZ project deals with ingesting LINZ documents. It relies on Tesseract and GDAL for document processing. Regular data reorganization is needed before running the scripts.

Gazette

The Gazette project uses Python requests to interact with the Gazette API. Ensure API keys are up to date and handle errors gracefully.

Waterford

This project handles web scraping of Waterford documents and relies on Selenium and Geckodriver. Ensure proper handling of login sessions and data extraction procedures.

Travel Today

The Travel Today project automates the collection of travel data from emails. The script parses email content to extract relevant travel information for further analysis.
